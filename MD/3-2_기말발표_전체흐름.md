# 3-2 기말발표 전체 흐름 정리

## 문서 목적

이 문서는 이번 `3-2 기말발표`를 준비하기 위한 발표 마스터 플로우 문서다.  
목표는 다음 세 가지다.

- 이번 발표의 전체 서사를 한 줄로 정리한다.
- 슬라이드별로 무엇을 말해야 하는지, 왜 그 순서가 맞는지 정리한다.
- 내가 위 대화에서 정리한 발표 방향을 유지하되, 실제 문서와 코드 검토를 바탕으로 넣으면 좋은 내용을 추가한다.

이 문서는 `product-manager` 스킬 관점으로 작성한다.  
즉, 발표를 단순히 예쁘게 만드는 것이 아니라 `무엇을 강조해야 청중이 제품의 가치와 현재 진척을 정확히 이해하는가`에 초점을 맞춘다.

---

## 0. PM 재검토 결과

냉정하게 보면, 기존 버전은 `방향성은 좋지만 아직 완벽한 발표 계획서라고 보기는 어렵다.`  
이유는 아래 네 가지다.

- 발표 가이드라인의 필수 항목이 슬라이드와 1:1로 매핑되어 있지 않았다.
- `핵심 기술`이 별도 메시지로 드러나지 않아 평가자 입장에서 누락처럼 보일 수 있었다.
- `향후 계획`이 방향 중심이었지, 가이드라인이 요구한 `일정 + 산출물` 형태로 충분히 구조화되지 않았다.
- `AI agent / MCP / skills`는 강한 차별점이지만, 현재 문서에서는 아직 교수나 심사자가 이해할 수 있는 언어로 충분히 번역되지 않았다.

즉, 좋은 문서였지만 `발표용 설계 문서`라기보다 `좋은 전략 메모`에 가까웠다.  
이번 수정에서는 그 차이를 메꾼다.

### 이번 수정의 원칙

- 발표 가이드라인의 필수 항목이 슬라이드 구조 안에서 빠짐없이 보이게 한다.
- `이번에 강조하고 싶은 것`은 살리되, `평가자가 반드시 확인하는 것`을 먼저 충족한다.
- 내부 용어는 결과 중심 언어로 번역한다.
- 발표 8분 내에 실제로 소화 가능한 흐름으로 다시 정리한다.

## 1. 이번 발표의 핵심 전략

## 1-1. 발표의 성격

이번 발표는 `아이디어 설명 발표`가 아니라 `구현 진척과 개발 방식의 설득 발표`가 되어야 한다.

지난 발표가 아래에 가까웠다면,

- 왜 이 문제가 중요한가
- 왜 이 서비스가 필요한가
- 왜 우리 구조가 다른가

이번 발표는 아래에 가까워야 한다.

- 그래서 우리는 현재 어디까지 만들었는가
- 그 기능들은 어떤 흐름으로 연결되는가
- 그리고 우리는 어떤 개발 방식으로 빠르게 실험하고 고도화하고 있는가

즉, 이번 발표의 무게중심은 `문제 정의`에서 `실제 구현 + 개발 운영 방식`으로 이동해야 한다.

## 1-2. 이번 발표에서 청중 머릿속에 남겨야 할 3가지

이번 발표가 끝났을 때 청중이 기억해야 할 문장은 아래 세 가지다.

- `PULSE는 외식업 사장님의 마케팅을 자동화하는 플랫폼이다.`
- `핵심 기능은 이미 실제로 구현되고 있다.`
- `이 팀은 AI agent, MCP, skills를 활용해 빠르게 실험하고 개선하는 방식으로 개발하고 있다.`

이 세 문장이 남으면 발표는 성공이다.

---

## 2. 발표의 한 줄 서사

이번 발표 전체를 한 문장으로 요약하면 다음과 같다.

`PULSE는 외식업 사장님의 마케팅 문제를 해결하는 자동화 플랫폼이며, 현재 핵심 기능을 구현해가고 있고, AI-native한 개발 방식으로 품질과 속도를 동시에 끌어올리고 있습니다.`

이 문장이 이번 발표의 SSOT가 되어야 한다.  
모든 슬라이드는 이 문장을 분해해서 설명하는 역할을 가져야 한다.

---

## 3. 발표에서 유지할 것과 바꿀 것

## 3-1. 유지할 것

지난 발표에서 반드시 가져와야 하는 자산은 아래와 같다.

- 프로젝트 한 줄 정의
  - `외식업 사장님의 마케팅 어려움을 해결하는 마케팅 자동화 플랫폼`
- 문제 구조 프레임
  - `지식 / 비용 / 실행`
- 구조적 차별화 메시지
  - `차별점은 기능이 아니라 구조입니다`
- 브랜드 카피
  - `외식업의 맥박을 읽다. 성장의 리듬을 만들다`

이 네 가지는 이미 좋은 발표 자산이다.  
버리기보다 더 압축해서 써야 한다.

## 3-2. 바꿔야 할 것

이번 발표에서는 아래가 반드시 달라져야 한다.

- 문제 정의 비중을 줄인다.
- 개발 완료 범위를 훨씬 앞쪽에 배치한다.
- 기능 소개를 `현재 구현 상태` 중심으로 재배열한다.
- 기술 설명을 `핵심 기술`과 `개발 방식`으로 나눈다.
- AI 영상 제작을 단순 기능 소개가 아니라 `프롬프트 실험을 통한 품질 고도화`로 설명한다.
- `향후 계획`은 반드시 `일정 + 산출물` 형태로 정리한다.
- 발표 시간 8분 안에 들어오도록 `시연 영상` 운영 방식을 포함해 구조를 다시 짠다.

## 4. 발표 가이드라인 대응표

이번 발표는 `내가 강조하고 싶은 것`보다 먼저 `평가 기준을 완벽히 충족하는 것`이 우선이다.  
따라서 아래처럼 필수 항목을 발표 구조와 직접 연결해야 한다.

| 가이드라인 필수 항목 | 발표에서 어디서 다룰지 | 비고 |
|---|---|---|
| 프로젝트명 | 1장 | 필수 |
| 팀명 | 1장 | 필수 |
| 팀원 이름 및 역할 | 1장 | 필수 |
| 프로젝트 개요 | 2장 | 필요성 + 기획 + 핵심 기술까지 포함 |
| 개발 진행 현황(3-2 발표 이후) | 3장 | 이번 발표의 핵심 |
| 향후 계획(일정 + 산출물) | 6장 | 일정과 결과물을 같이 보여줘야 함 |
| 기대 효과 | 7장 | 너무 추상적이면 안 됨 |

즉, `개발 방식`은 매우 좋은 차별점이지만 필수 항목은 아니다.  
그래서 이 요소는 `필수 항목을 다 채운 뒤, 가장 강한 차별화 요소로 넣어야` 한다.

## 5. 최종 권장 흐름

가이드라인과 실제 발표 시간을 동시에 고려했을 때, 가장 현실적인 구조는 `7장 + 시연영상 삽입`이다.

### 추천 구성

1. 프로젝트 기본 정보
2. 프로젝트 개요: 필요성 + 기획 + 핵심 기술
3. 개발 진행 현황
4. 핵심 기능 연결 구조 + 짧은 시연영상
5. 우리의 개발 방식
6. 향후 계획: 일정 + 산출물
7. 기대 효과 + 마무리

### 이 구조가 더 좋은 이유

- 2장에서 가이드라인이 요구한 `프로젝트 개요(필요성 + 기획 + 핵심 기술)`를 한 번에 처리할 수 있다.
- 3장에서 `지난 발표 이후 무엇이 달라졌는가`를 명확히 보여줄 수 있다.
- 4장에서 기능 설명과 시연을 결합해 시간을 절약할 수 있다.
- 5장에서 네가 가장 강조하고 싶은 `AI-native 개발 방식`을 자연스럽게 차별점으로 밀 수 있다.
- 6장에서 평가자가 좋아하는 `일정 + 산출물` 구조를 명시할 수 있다.

### 권장 시간 배분

- 1장: 35초
- 2장: 1분 15초
- 3장: 1분 20초
- 4장: 1분 20초 + 시연영상 40초
- 5장: 1분 20초
- 6장: 55초
- 7장: 35초

총 7분 45초 내외

이 정도면 Q&A 2분을 남기기 좋다.

## 6. 슬라이드별 상세 흐름

## 6-1. 1장 프로젝트 기본 정보

### 목적

발표 시작 30~40초 안에 프로젝트 정체성과 팀 구성을 명확히 전달한다.

### 꼭 들어갈 내용

- 프로젝트명: PULSE
- 팀명: F12
- 팀원 이름 및 역할
- 프로젝트 한 줄 소개

### 추천 핵심 문장

`저희는 외식업 사장님의 마케팅 어려움을 해결하는 지능형 마케팅 자동화 플랫폼 PULSE를 만들고 있습니다.`

### 발표용 확정 표기안

- 노태경 - PO/기획 총괄, UX/UI, 프론트엔드
- 김혜린 - UX/UI, 프론트엔드
- 오해서 - 백엔드, AI
- 윤준하 - 개발 총괄, 백엔드, AI

### 이 장에서 중요한 포인트

- 역할은 `실제 기여 축` 중심으로 보여주는 것이 좋다.
- 발표에서는 이름을 읽고 끝내지 말고 `이 팀이 제품-디자인-프론트-백엔드-AI를 모두 커버하는 팀`이라는 인상을 줘야 한다.

### 발표 톤

짧고 명확하게.  
여기서 설명을 길게 하면 이후 시간이 부족해진다.

## 6-2. 2장 프로젝트 개요: 필요성 + 기획 + 핵심 기술

### 목적

가이드라인의 `프로젝트 개요(필요성 + 기획 + 핵심 기술)`를 한 장에서 명확히 충족한다.

### 꼭 들어갈 내용

- 필요성
  - 외식업 사장님의 마케팅 문제
  - 3가지 장벽: 지식 / 비용 / 실행
- 기획
  - PULSE가 어떤 루프로 문제를 해결하는지
- 핵심 기술
  - 리뷰/상권 분석
  - AI 기반 영상 생성
  - 리뷰 답변 자동화
  - React + Spring Boot + FastAPI + Kakao API + VEO3 정도의 상위 수준 기술 언급
  - 차별화 초안: DSPy 및 AI 활용 방식은 별도 차별점 메시지로 짧게 예고

### 추천 핵심 문장

`PULSE는 외식업 사장님의 지식·비용·실행 장벽을 줄이기 위해, 분석과 생성, 실행을 하나의 흐름으로 연결한 마케팅 자동화 플랫폼입니다.`

### 발표 포인트

이 장에서 빠지면 안 되는 것이 바로 `핵심 기술`이다.  
기존 문서는 문제 정의가 강했지만, 가이드라인 기준으로 보면 `핵심 기술`이 약하게 보일 수 있었다.  
따라서 이번에는 다음처럼 한 줄로 기술을 정리하는 것이 좋다.

`저희는 리뷰/상권 분석, AI 영상 생성, 리뷰 답변 자동화를 중심으로 React 프론트엔드와 Spring Boot, FastAPI 기반 구조를 설계했습니다.`

### AI 파트 초안 메모

이 장에서는 기존 기술스택을 유지하되, `DSPy와 AI 활용 방식`은 너무 깊게 설명하지 말고 아래 수준으로 예고만 하는 것이 좋다.

발표용 초안:

`기존 서비스 기술 스택은 유지하되, AI 파트에서는 프롬프트 구조화와 품질 개선을 위해 DSPy 및 다양한 AI 활용 방식을 함께 실험하고 있습니다.`

이 문장은 AI 담당 팀원이 이후 더 기술적으로 다듬으면 된다.

### 슬라이드 구성 팁

- 상단: 문제 정의 1줄
- 중단: 3대 장벽
- 하단: 해결 루프 + 핵심 기술 3개

한 장에 넣되, 텍스트를 늘리지 말고 도식 중심으로 가야 한다.

## 6-3. 3장 현재 개발 진행 현황

### 목적

이번 발표의 첫 번째 핵심 장이다.  
이제는 실제 구현이 진행되고 있음을 명확하게 보여준다.

### 반드시 포함할 현재 상태

- 개발 완료
  - 주변 상권 분석
  - 손님 분석
  - 홍보영상 만들기
  - 리뷰 관리 답변
- UI 구현 완료
  - 인플루언서 매칭

### 추천 핵심 문장

`저희는 아이디어 제안 단계를 넘어, 주변 상권 분석과 손님 분석, 홍보영상 만들기, 리뷰 관리 답변 기능의 구현을 완료했고, 인플루언서 매칭은 UI 구현을 완료한 상태입니다.`

### 발표용 상태 표현 확정안

- 주변 상권 분석: 완료
- 손님 분석: 완료
- 홍보영상 만들기: 기능 구현 완료, 프롬프트 테스트 진행 중
- 리뷰 관리 답변: 완료
- 인플루언서 매칭: UI 구현 완료

### 슬라이드 표현 방식 추천

- `완료 / 구현 중 / 다음 단계` 3단 레이아웃
- 또는 기능별 체크리스트
- 가능하면 `지난 발표 대비` 표시 추가

예시:

- 완료: 손님 분석, 주변 상권 분석, 리뷰 관리 답변
- 완료(테스트 진행 중): 홍보영상 만들기
- 완료(UI): 인플루언서 매칭
- 다음 단계: 연동 고도화, 실사용 검증, 매칭 기능 고도화

### PM 관점 보완점

이 장은 단순한 현황 나열보다 `3-2 발표 이후 무엇이 진전되었는가`로 읽혀야 한다.  
따라서 가능하면 다음 비교 프레임을 추천한다.

- 지난 발표: 핵심 기능 방향 제시
- 이번 발표: 핵심 기능 구현 완료 + 확장 기능 UI 완료

이 비교가 들어가면 청중은 팀의 진척을 더 선명하게 느낀다.

### 왜 중요한가

이번 발표의 평가 포인트 중 하나는 `지금 어디까지 왔는가`다.  
따라서 이 장은 절대 뒤로 밀리면 안 된다.

## 6-4. 4장 핵심 기능 연결 구조 + 시연영상

### 목적

기능이 따로 노는 것이 아니라 하나의 루프로 연결된다는 점을 보여준다.

### 추천 구조

- 손님 분석
- 주변 상권 분석
- 홍보영상 만들기
- 리뷰 관리 답변
- 인플루언서 매칭(Pro / 확장)

### 추천 핵심 문장

`저희의 차별점은 기능 하나하나가 아니라, 분석 결과가 실행으로 이어지고 다시 다음 행동으로 연결되는 구조에 있습니다.`

### 여기서 강조할 메시지

지난 발표에서 강했던 메시지인 `차별점은 기능이 아니라 구조입니다`를 그대로 가져오되, 이번에는 구현 상태와 연결한다.

예시 흐름:

- 손님 분석으로 누구에게 팔지 정한다.
- 주변 상권 분석으로 우리 위치와 경쟁 맥락을 본다.
- 그 인사이트를 바탕으로 홍보 영상을 만든다.
- 유입 이후에는 리뷰 답변으로 고객 관계를 관리한다.
- 더 확장하면 인플루언서 매칭으로 외부 채널까지 넓힌다.

### 코드 검토 기반으로 넣으면 좋은 추가 포인트

코드상 실제로 이 연결 구조가 드러난다.

- `UnifiedInsightPage`는 손님 분석 데이터를 불러와 페르소나를 선택하고, 이후 영상 제작 흐름으로 연결되는 구조를 갖고 있다.
- `PromotionPage`는 분석 데이터에서 불러온 페르소나를 바탕으로 자동 프롬프트를 생성한다.
- `StatusV2Page`의 AI 제안 카드는 바로 영상 제작 액션으로 연결된다.

즉, 발표에서는 단순히 `연결됩니다`라고 말하는 것이 아니라  
`실제 UI 흐름도 그렇게 설계돼 있습니다`라고 말할 수 있다.

### 시연영상 운영 방식

기존 문서에서 빠져 있던 가장 큰 실무 포인트가 바로 이것이다.  
발표 시간이 `발표 + 시연영상 포함 8분`에 가깝기 때문에, 시연은 별도 섹션이 아니라 이 장 안에 흡수해야 한다.

네 답변 기준으로는 모든 기능을 mock 기반으로 보여줄 수 있다.  
하지만 PM 관점에서 추천하는 시연은 `모든 기능을 다 보여주는 방식`이 아니라 `핵심 구조가 가장 잘 보이는 흐름` 위주다.

추천 방식:

- 20초: 손님 분석 또는 상권 분석 화면
- 20초: 분석 결과 기반 영상 생성 흐름

추가 대안:

- 리뷰 관리 답변은 발표 시간 여유가 있으면 10~15초짜리 보조 컷으로 삽입
- 인플루언서 매칭은 `UI 구현 완료` 증빙 화면으로만 짧게 보여주고 설명은 길게 하지 않기

핵심은 `기능 개수`보다 `연결 구조`가 보이게 하는 것이다.

즉, 시연도 아래 순서를 지켜야 한다.

- 분석 결과 확인
- 바로 생성 액션으로 이동
- 결과물이 만들어지는 흐름 확인

이렇게 해야 발표 메시지와 시연 메시지가 일치한다.

## 6-5. 5장 이번 발표의 핵심: 우리의 개발 방식

### 목적

이번 발표의 두 번째 핵심 장이다.  
기능 구현을 넘어서, 팀이 어떤 방식으로 개발하고 있는지 보여준다.

### 추천 핵심 문장

`저희는 AI를 사용자 기능에만 쓰는 것이 아니라, 개발 프로세스 자체도 AI-native하게 운영하고 있습니다.`

### 반드시 들어갈 내용

- AI agent 활용
- MCP 활용
- skills 활용
- DSPy 및 AI 활용 방식 초안
- 반복 실험과 빠른 수정

### 설명 방향

추상적으로 말하면 안 된다.  
반드시 아래처럼 `개발 성과`와 연결해야 한다.

- AI agent를 활용해 작업을 잘게 나누고 빠르게 탐색했다.
- MCP를 통해 필요한 도구와 문맥을 연결하며 작업 전환 비용을 줄였다.
- skills를 통해 반복 작업과 문서화, 품질 기준을 구조화했다.

### 코드/문서 검토 기반 근거

프로젝트 안에는 이미 이 메시지를 뒷받침하는 흔적이 있다.

- `.agent/skills/README.md`
  - 프로젝트가 역할별 스킬 체계로 운영되고 있음을 보여준다.
- `MD/CHANGELOG.md`
  - Agent Skills System Migration이 기록되어 있다.
- `MD/PULSE.md`
  - AI agent가 프로젝트를 이해하고 개발할 수 있도록 SSOT 성격의 지식 문서가 정리돼 있다.

### AI 담당 파트 초안

이 장은 최종적으로 AI 담당 팀원이 보강하면 가장 좋다.  
다만 현재 발표 흐름용 초안은 아래처럼 잡는 것을 추천한다.

- 문서 정리: 기능과 구조를 문서화해 AI가 동일한 맥락으로 작업할 수 있게 함
- 기능 기획: 기능 요구사항과 우선순위 정리에 AI 활용
- 코드 탐색: 기존 파일 구조와 흐름 파악에 AI 활용
- 코드 제작: 반복 UI/로직 작업의 초안 생산에 AI 활용
- 반복 작업 자동화: 반복 수정, 검토, 정리 과정 보조
- Git: 변경 범위 정리와 작업 흐름 관리 보조
- DSPy 초안: 프롬프트 품질을 구조적으로 개선하거나 실험하는 방향으로 활용 가능

발표용 초안 문장:

`저희는 AI를 단순 생성 도구로만 쓰지 않고, 문서 정리, 기능 기획, 코드 탐색과 제작, 반복 작업 자동화, Git 작업 보조까지 개발 전반에 활용하고 있습니다. 또한 DSPy 기반 접근도 프롬프트 품질과 구조를 더 안정적으로 만들기 위한 방향으로 검토하고 있습니다.`

### PM 관점에서 반드시 수정해야 하는 표현

기존 문서의 약점은 `AI agent / MCP / skills`라는 용어 자체가 중심에 너무 가까이 있었다는 점이다.  
심사자는 이 용어보다 `그래서 뭐가 좋아졌는가`를 더 궁금해한다.

따라서 이 장에서는 용어를 먼저 말하지 말고 결과를 먼저 말해야 한다.

추천 순서:

- 개발 실험 속도를 높였다.
- 반복 작업의 일관성을 높였다.
- 문서와 역할을 구조화해 협업 효율을 높였다.
- 그 방법으로 AI agent / MCP / skills를 사용했다.

즉, `도구 -> 효과`가 아니라 `효과 -> 도구` 순서로 말하는 게 훨씬 낫다.

### 이 장에서 넣으면 좋은 추가 문장

`즉, 저희는 개발을 사람의 감에만 맡기지 않고, 에이전트가 따라올 수 있는 문서와 절차를 함께 만들면서 팀의 개발 속도와 일관성을 높이고 있습니다.`

이 문장은 발표에서 상당히 강하게 먹힐 가능성이 높다.

|---|---|---|

이번에는 반드시 `언제까지 무엇을 만들고 검증할지`가 보이게 해야 한다.

또한 이 장에서 `AI 영상 품질 개선을 위한 프롬프트 실험 지속`을 산출물 중 하나로 넣는 것은 매우 좋다.

## 6-7. 7장 기대 효과 + 마무리

### 목적

제품과 팀의 가치를 넓은 관점에서 정리하고, 발표를 단단하게 마무리한다.

### 추천 핵심 문장

`PULSE는 외식업 사장님의 마케팅 실행 부담을 줄이고, 데이터 기반 의사결정을 돕는 동시에, 지역 상권의 디지털 전환을 돕는 방향으로 확장될 수 있습니다.`

### 넣을 수 있는 효과

- 사장님의 마케팅 실행 부담 감소
- 콘텐츠 제작 시간과 진입장벽 완화
- 리뷰 응대 시간 단축 및 응답 일관성 향상
- 데이터 기반으로 다음 행동을 결정할 수 있는 운영 보조
- 장기적으로는 지역 상권의 디지털 전환 보조

### PM 권장 기대 효과 톤

기대 효과는 너무 과장하면 안 된다.  
이번 발표에서는 아래처럼 `직접 효과 -> 확장 효과` 순서로 말하는 것이 가장 안전하다.

- 직접 효과
  - 사장님이 마케팅 실행을 시작하기 쉬워진다
  - 분석 결과를 바로 행동으로 연결할 수 있다
  - 리뷰 응대와 콘텐츠 제작 시간을 줄일 수 있다
- 확장 효과
  - 외식업 소상공인의 디지털 마케팅 진입장벽을 낮출 수 있다
  - 지역 상권 단위의 디지털 격차 완화 가능성이 있다

### 이 장에서 넣으면 좋은 마무리

`PULSE는 외식업의 맥박을 읽고, 사장님의 성장 리듬을 만드는 서비스가 되고자 합니다.`

이 문장으로 닫으면 브랜드 톤이 살아난다.

## 7. 별도 강조 포인트: AI 영상 품질 고도화 실험

기존 문서에서는 이 내용을 하나의 별도 슬라이드로 뒀지만, 냉정하게 보면 8분 발표에서 독립 슬라이드로 쓰기에는 우선순위가 애매하다.

즉, `독립 슬라이드`보다 `팀 역량을 증명하는 사례`로 쓰는 편이 더 강하다.

### 이 내용을 발표에서 말하는 방식

`특히 홍보영상 기능에서는 단순히 모델을 붙이는 데서 끝나지 않고, 어떤 타겟과 무드, 장면 설명을 어떤 구조로 전달해야 더 높은 품질의 결과가 나오는지 반복적으로 실험하고 있습니다.`

### 코드/문서 기반 근거

  - VEO3용 프롬프트 명세가 별도로 존재한다.

  - `generateVeoPayload` 함수가 있다.
- `src/features/promotion/PromotionPage.jsx`
  - 분석 결과 기반 자동 프롬프트 추천 흐름이 있다.

이건 여전히 매우 좋은 차별화 포인트다.  
다만 `별도 기능 소개`가 아니라 `팀의 학습 능력과 품질 개선 방식`으로 위치시키는 것이 더 강하다.

## 8. 코드 검토를 통해 이번 발표에 추가하면 좋은 내용

위 대화에서 정리한 방향 외에, 코드 검토를 통해 발표에 넣으면 좋은 내용은 아래와 같다.

## 6-1. `분석 결과에서 바로 실행으로 연결되는 UI 흐름`

이건 발표에 꼭 넣는 것이 좋다.  
왜냐하면 서비스의 강점이 `AI 분석` 자체보다 `분석 후 무엇을 하게 만드는가`에 있기 때문이다.

근거:

- `PromotionPage.jsx`는 분석 데이터와 선택한 페르소나를 받아 자동 프롬프트를 구성한다.
- `StatusV2Page.jsx`의 AI 제안 카드는 바로 `promotion`으로 액션을 연결한다.
- `UnifiedInsightPage.jsx`는 손님 분석 데이터 로드 후 특정 페르소나를 선택하게 설계되어 있다.

발표용 표현:

`저희는 분석 결과를 보여주는 데서 멈추지 않고, 그 결과를 바로 다음 행동으로 연결하는 흐름까지 설계했습니다.`

## 6-2. `주변 상권 분석은 단순 목업이 아니라 실제 데이터 연동 의도가 강한 기능`

근거:

- `CommercialAnalysisPage.jsx`는 Kakao Places API 기반 실시간 조회 구조를 갖고 있다.
- 반경별 조회, 경쟁 업소 리스트, 지도 이동, 검색 흐름이 있다.

발표용 표현:

`주변 상권 분석은 단순한 정적 화면이 아니라, 실제 지역 데이터와 지도를 결합해 주변 업소 맥락을 읽을 수 있도록 구성했습니다.`

## 6-3. `리뷰 관리 답변 기능은 생각보다 완성도가 높다`

이 기능은 단순히 “리뷰 답변 생성”으로만 소개하면 아쉽다.

근거:

- `ReviewManagementPage.jsx`는
  - 리뷰 요약
  - 실제 리뷰 리스트
  - 빠른 설정
  - 저장 템플릿
  - 답변 재생성
  - 설정 자동 저장
  구조를 가지고 있다.

발표용 표현:

`리뷰 관리 답변 기능은 단순 초안 생성에 그치지 않고, 답변 톤 설정, 예외 케이스 반영, 템플릿 저장과 재사용까지 포함한 운영형 기능으로 확장했습니다.`

이건 꽤 좋은 포인트다.  
발표에서 기능의 깊이를 보여준다.

## 6-4. `인플루언서 매칭은 UI만 구현 완료됐지만, 구조는 이미 꽤 구체적이다`

근거:

- `InfluencerMatchingPage.jsx`는
  - 검색
  - 카테고리 필터
  - AI 매칭 점수 정렬
  - 상세 모달
  - 제안 보내기 흐름
  을 갖고 있다.
- `App.jsx`에는 인플루언서 대시보드와 제안 수락 라우트도 분리돼 있다.

발표용 표현:

`인플루언서 매칭은 현재 UI 구현을 완료했고, 추천 탐색, 상세 확인, 제안 연결까지의 사용자 흐름을 구체적으로 설계해 둔 상태입니다.`

## 6-5. `대시보드가 행동 중심으로 설계돼 있다는 점`

지난 발표의 핵심 메시지와도 맞닿는 포인트다.

근거:

- `MD/PULSE.md`에 `Action-Centric`, `Single Loop`, `No Scroll Policy`가 명시돼 있다.
- `StatusV2Page.jsx`는 “오늘의 우리 가게 요약”, “지금 당장 해볼까요?” 같은 행동 중심 카드를 전면에 둔다.

발표용 표현:

`저희는 숫자를 많이 보여주는 대시보드보다, 사장님이 바로 행동할 수 있게 만드는 행동 중심 대시보드를 지향하고 있습니다.`

이건 발표에서 꽤 세련된 메시지다.

## 9. 이번 발표에서 추천하는 강조 순서

이번 발표에서는 아래 순서대로 감정을 쌓는 것이 좋다.

### 1단계. 공감

- 외식업 사장님은 마케팅을 하고 싶어도 어렵다.

### 2단계. 신뢰

- 이 문제는 인터뷰와 조사로 확인했다.

### 3단계. 진척

- 핵심 기능은 실제로 구현되고 있다.

### 4단계. 구조

- 기능은 따로 있는 것이 아니라 루프로 연결된다.

### 5단계. 팀 역량

- 우리는 AI agent, MCP, skills를 활용해 빠르게 만들고 있다.

### 6단계. 확장성

- 앞으로는 기능 연결과 실사용 검증을 더 고도화할 계획이다.

이 순서가 좋은 이유는,  
청중이 `문제 공감 -> 제품 신뢰 -> 구현 신뢰 -> 팀 신뢰` 순서로 발표를 받아들이기 때문이다.

## 10. 발표 중 실제로 사용할 만한 연결 문장

## 8-1. 개요 -> 개발 현황 전환

`이 문제를 해결하기 위해 저희는 단순 아이디어 제안이 아니라, 실제로 기능 구현을 진행해 왔습니다.`

## 8-2. 개발 현황 -> 구조 전환

`그리고 중요한 것은 이 기능들이 따로 존재하는 것이 아니라, 하나의 실행 흐름으로 연결된다는 점입니다.`

## 8-3. 구조 -> 개발 방식 전환

`여기서 이번 발표에서 특히 강조드리고 싶은 부분은, 저희가 이 기능들을 어떤 방식으로 만들어 왔는가입니다.`

## 8-4. 개발 방식 -> 영상 실험 전환

`특히 홍보영상 기능에서는 단순 구현을 넘어서, 실제 결과 품질을 높이기 위한 프롬프트 실험을 계속하고 있습니다.`

## 8-5. 향후 계획 전환

`이제 남은 과제는 기능을 더 많이 추가하는 것보다, 지금 구현한 구조를 더 정교하게 연결하고 검증하는 것입니다.`

## 12. 이번 발표에서 절대 놓치면 안 되는 리스크

### 1) `개발 완료`를 과장해서 말하면 안 된다

특히 인플루언서 매칭은 `UI 구현 완료`라고 분명히 선을 그어야 한다.  
AI 매칭, 제안 흐름, 대시보드 구조가 있어도 백엔드/실운영까지 완료된 것처럼 말하면 오히려 신뢰를 잃는다.

### 2) `AI agent / MCP / skills`를 도구 자랑처럼 말하면 안 된다

발표에서는 용어보다 결과를 말해야 한다.

- 잘못된 방식: `저희는 MCP를 썼습니다`
- 더 좋은 방식: `저희는 반복 작업과 탐색을 빠르게 하기 위해 AI agent와 도구 연결 방식을 적극 활용했습니다`

### 3) `핵심 기술`을 빠뜨리면 안 된다

이번 발표에서 가장 쉽게 놓치는 항목이 바로 이것이다.  
문제 정의와 개발 방식에 집중하다가, 정작 가이드라인이 요구한 `핵심 기술`이 흐릿해질 수 있다.

### 4) 시연은 기능 나열이 아니라 연결 구조를 보여줘야 한다

시연이 `화면 3개 보여주기`로 끝나면 약하다.  
반드시 `분석 -> 생성` 혹은 `분석 -> 제안` 흐름을 보여줘야 한다.

## 14. 발표 준비 실무 체크리스트

- 프로젝트 기본 정보 슬라이드에 팀원 역할을 명확히 넣기
- 프로젝트 개요 슬라이드에 `필요성 + 기획 + 핵심 기술`이 모두 들어가게 하기
- 문제 정의는 1장으로 압축하기
- 개발 진행 현황 슬라이드를 앞쪽에 배치하기
- 완료/진행 중/예정 상태를 절대 섞지 않기
- 기능 소개는 `연결 구조` 중심으로 말하기
- 시연영상은 40초 안팎으로 짧게 넣고, 연결 구조를 보여주기
- AI agent, MCP, skills는 `왜 썼는지`와 `무슨 효과가 있었는지`를 함께 말하기
- 영상 프롬프트 실험은 `품질 고도화 과정`으로 설명하기
- 향후 계획은 반드시 `일정 + 산출물` 표로 정리하기

이 기준만 유지하면 이번 발표의 흐름은 안정적으로 잡힌다.
